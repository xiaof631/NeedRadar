# 需求文档：NeedRadar RSS 需求挖掘系统

## 1. 背景与目标
NeedRadar 旨在从互联网上公开可访问的 RSS 数据源中自动挖掘潜在的“小工具”开发需求。系统需要构建一条从原始信息收集、过滤分析到需求产出的完整流水线，以支持后续的立项与深入调研。

## 2. 术语定义
- **原始内容**：从各类 RSS 源（如 Reddit Subreddit、技术博客、工具分享站等）中抓取的条目，包含标题、摘要、正文、链接、作者、发布时间等字段。
- **候选需求**：经由规则和 LLM 分析后判定具有开发潜力的需求描述，包含需求要点和初步价值评估。
- **需求挖掘流水线**：RSS 抓取 → 原始内容库 → 候选需求库 → （可选）LLM 结构化处理 → 立项 → 深挖 → 小工具开发。

## 3. 用户与场景
- **需求分析师/产品经理**：关注最新可实现的小工具创意，依赖系统持续输出可行的需求列表。
- **开发者/独立创造者**：基于候选需求快速验证并实现 MVP。
- **运营人员**：监控需求挖掘成果，为后续推广与内容运营提供素材。

典型场景：系统每日自动拉取多个 RSS 源，筛选出具有开发潜力的创意，并将结构化需求同步至内部管理平台或发给后续的深挖团队。

## 4. 功能需求
### 4.1 RSS 数据采集
1. 支持配置多种公开 RSS 源，包含名称、URL、类型（社区、博客、工具站等）、抓取频率。
2. 支持增量抓取，避免重复收录已处理的条目。
3. 捕获并存储条目关键字段：标题、摘要/内容、发布时间、链接、标签/分类、作者。
4. 针对抓取失败提供重试与失败告警日志。

### 4.2 原始内容库管理
1. 将抓取到的条目保存至持久化存储（例如 SQLite/PostgreSQL），包含抓取时间与数据源信息。
2. 提供基本的状态标记：未处理、已筛选、已入库候选需求、忽略等。
3. 支持按时间、数据源、关键词等查询与导出。

### 4.3 候选需求筛选
1. 配置式的规则过滤：关键词匹配、数据源白名单/黑名单、文本长度阈值等。
2. 支持 LLM 辅助判定是否具备可开发的小工具潜力（可选择开源 LLM 或本地部署模型，无需 API Key）。
3. 支持为每条候选需求生成结构化信息：问题描述、目标用户、潜在价值、竞争情况初评。
4. 对筛选过程生成评估指标（如通过率、来源分布、每源命中率）。

### 4.4 候选需求库管理
1. 将筛选后的候选需求入库，保留原始条目引用与生成的结构化信息。
2. 支持状态流转：待复核、已立项、拒绝、待深挖等。
3. 提供检索与过滤能力，支持导出 JSON/CSV 供其他系统使用。
4. 支持对候选需求进行手动标注与补充说明。

### 4.5 流水线调度与监控
1. 支持定时任务调度（例如 cron 表达式）驱动整条流水线运行。
2. 提供运行日志与错误告警，便于定位问题。
3. 可视化基本统计（抓取量、筛选通过量、转化率等）。

### 4.6 与需求挖掘系统的集成
1. 候选需求库需支持导出与下游“需求挖掘系统”对接的接口（REST/GraphQL 或文件同步）。
2. 提供 webhook 或消息队列推送新入库的候选需求。
3. 为后续深挖阶段保留充足上下文（原始链接、摘要、结构化字段）。

## 5. 非功能需求
1. **合法性**：仅使用公开无需授权的 RSS 源，不引入需要 API Key/OAuth 的私有接口。
2. **可扩展性**：易于新增 RSS 源与筛选规则，支持插件化扩展。
3. **可靠性**：在网络波动时具备重试与容错能力，确保任务不中断。
4. **可维护性**：代码模块化，提供清晰文档与配置示例。
5. **安全性**：隔离存储敏感信息（如内部配置），对外接口需鉴权。

## 6. 里程碑
1. **MVP 阶段**：实现 RSS 抓取、原始内容入库、简单规则筛选、候选需求入库及导出。
2. **结构化增强阶段**：引入 LLM 分析与结构化生成、完善状态流转与统计。
3. **集成阶段**: 搭建与需求挖掘系统的自动对接、完善监控与告警。

## 7. 未来展望
- 引入更多数据源类型（如 Twitter 列表、播客 RSS 等）并通过插件机制接入。
- 使用自监督学习从历史标注中持续优化筛选模型。
- 构建需求知识图谱，支持关联分析与趋势预测。
