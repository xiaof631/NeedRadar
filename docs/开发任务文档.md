# 开发任务文档：NeedRadar RSS 需求挖掘系统

## 1. 目标概述
本任务旨在依据《开发文档：NeedRadar RSS 需求挖掘系统》实施整体研发工作，分阶段完成 RSS 采集、原始内容管理、筛选分析、候选需求管理以及调度与监控能力的建设，确保系统具备端到端的数据流转与可运维性。

## 2. 里程碑与交付物
| 里程碑 | 时间预估 | 关键交付物 |
| --- | --- | --- |
| M1 环境与基础设施 | 第 1 周 | ✅ 本地与测试环境、数据库迁移框架、基础 Docker/配置骨架 |
| M2 采集层实现 | 第 2-3 周 | ✅ RSS 源管理 API/CLI、抓取器、重试与日志记录 |
| M3 原始内容与筛选 | 第 4-5 周 | ✅ 原始内容 API、规则引擎、LLM 调用适配层 |
| M4 候选需求与调度 | 第 6-7 周 | ✅ 候选需求全流程、调度任务、导出/同步机制 |
| M5 测试与监控上线 | 第 8 周 | ✅ 自动化测试、监控指标、告警策略、CI/CD 打通 |

## 3. 详细任务拆解
### 3.1 环境与基础设施
- 建立 FastAPI 项目结构，配置 Typer/Click 命令行框架。
- 集成 SQLAlchemy、Alembic，配置 SQLite（开发）与 PostgreSQL（生产）连接。
- 提供 Docker 与 docker-compose 模板，抽象环境变量配置。
- 集成 structlog、OpenTelemetry 基础配置。
- 建立 GitHub Actions 工作流，执行 pytest、ruff、mypy、Docker 构建。

### 3.2 RSS 源管理与采集层
- 设计 `rss_sources`、`fetch_logs` ORM 模型与迁移脚本。
- 实现 RSS 源 CRUD API 与 CLI，支持频率、分类等配置管理。
- 编写调度任务 `fetch_rss`：
  - 调度器根据频率派发抓取任务。
  - 抓取器使用 `feedparser`，支持 ETag/Last-Modified 增量抓取。
  - 写入 `raw_entries`，生成 hash 去重。
- 设计失败重试策略与日志记录，并暴露监控指标。

### 3.3 原始内容层
- 设计 `raw_entries` ORM 模型与迁移。
- 实现原始条目查询、分页、状态更新、导出（CSV/JSON）API。
- 提供 CLI 工具导出与批量状态操作。
- 增加状态字段与索引优化查询性能。

### 3.4 筛选分析层
- 设计 `filter_rules` 模型与配置管理 API/CLI。
- 实现规则引擎：关键词、正则、评分阈值，输出 `rule_score`。
- 抽象 `LLMClient` 接口，支持多后端配置；实现基础适配器。
- 编排筛选流程：
  - 选取未处理 `raw_entries`。
  - 先运行规则过滤，通过后调用 LLM。
  - 生成结构化结果并存储至 `candidate_needs`。
- 为 LLM 调用加入超时、重试与监控指标。

### 3.5 候选需求层
- 设计 `candidate_needs` 模型与状态机字段。
- 实现候选需求的 REST API 与 CLI：查询、过滤、状态更新、备注。
- 构建导出/推送机制，支持 Webhook、CSV、消息队列。
- 增加状态流转日志与权限控制。

### 3.6 调度与集成层
- 使用 APScheduler 编写周期任务：`fetch_rss`、`run_filters`、`llm_analyze`、`sync_downstream`。
- 结合 Celery Worker 承载耗时任务（LLM、批量导出）。
- 集成 Redis/RabbitMQ 作为任务队列。
- 构建 Prometheus 指标导出、Grafana 仪表板配置。
- 设置异常告警（抓取失败率、队列积压等）。

### 3.7 测试与质量保障
- 单元测试：
  - RSS 抓取器增量逻辑、去重与重试。
  - 规则引擎匹配与评分。
  - LLM 客户端接口模拟。
- 集成测试：模拟从抓取到候选需求的流水线闭环。
- 覆盖率要求 ≥ 75%，配置 coverage 报告上传。
- 静态检查：ruff、mypy，保障类型正确性与编码规范。

### 3.8 安全、合规与运维
- 接口认证：Token/Key 校验，HTTPS/TLS 配置指南。
- 日志脱敏与访问控制，记录数据来源以便审计。
- 数据维护：原始内容归档策略、候选需求生命周期管理。
- 运维：定期清理任务、备份策略、容量规划建议。

## 4. 依赖与风险
- **外部依赖**：Redis/RabbitMQ、LLM 服务、Prometheus/Grafana。
- **风险点**：
  - LLM 调用稳定性与成本。
  - RSS 源质量与频率控制。
  - 多任务调度导致的资源竞争。
- **缓解措施**：
  - 对 LLM 调用实现熔断与降级策略。
  - 对 RSS 源设置采集限速与异常标记。
  - 对调度任务做优先级与资源隔离配置。

## 5. 资源与协作建议
- 团队配置：后端开发 2-3 人、运维/DevOps 1 人、数据分析顾问 1 人。
- 沟通机制：每周例会跟踪里程碑，使用 issue/看板同步任务。
- 文档维护：持续更新 API 文档、架构图、监控仪表板配置。

## 6. 验收标准
- 采集至候选需求全链路可在本地与测试环境跑通。
- 所有关键 API、CLI 均有文档与自动化测试覆盖。
- 监控指标在 Prometheus 中可观测，告警策略可触发测试。
- Docker 化部署脚本可一键启动主要服务。
