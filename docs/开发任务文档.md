# 开发任务文档：NeedRadar RSS 需求挖掘系统

## 1. 目标概述
本任务旨在依据《开发文档：NeedRadar RSS 需求挖掘系统》实施整体研发工作，分阶段完成 RSS 采集、原始内容管理、筛选分析、候选需求管理、Web 管理前端以及调度与监控能力的建设，确保系统具备端到端的数据流转与可运维性。

## 2. 里程碑与交付物
| 里程碑 | 时间预估 | 关键交付物 |
| --- | --- | --- |
| M1 环境与基础设施 | 第 1 周 | ✅ 本地与测试环境、数据库迁移框架、基础 Docker/配置骨架 |
| M2 采集层实现 | 第 2-3 周 | ✅ RSS 源管理 API/CLI、抓取器、重试与日志记录 |
| M3 原始内容与筛选 | 第 4-5 周 | ✅ 原始内容 API、规则引擎、LLM 调用适配层 |
| M4 候选需求与调度 | 第 6-7 周 | ✅ 候选需求全流程、调度任务、导出/同步机制 |
| M5 Web 管理前端 | 第 7-8 周 | ✅ 仪表盘、源管理、筛选监控、候选需求工作台、预留权限体系接口 |
| M6 测试与监控上线 | 第 9 周 | ✅ 自动化测试、监控指标、告警策略、CI/CD 打通 |

## 3. 详细任务拆解
### 3.1 环境与基础设施
- 建立 FastAPI 项目结构，配置 Typer/Click 命令行框架。
- 集成 SQLAlchemy、Alembic，配置 SQLite（开发）与 PostgreSQL（生产）连接。
- 提供 Docker 与 docker-compose 模板，抽象环境变量配置。
- 集成 structlog、OpenTelemetry 基础配置。
- 建立 GitHub Actions 工作流，执行 pytest、ruff、mypy、Docker 构建。

### 3.2 RSS 源管理与采集层
- 设计 `rss_sources`、`fetch_logs` ORM 模型与迁移脚本。
- 实现 RSS 源 CRUD API 与 CLI，支持频率、分类等配置管理。
- 编写调度任务 `fetch_rss`：
  - 调度器根据频率派发抓取任务。
  - 抓取器使用 `feedparser`，支持 ETag/Last-Modified 增量抓取。
  - 写入 `raw_entries`，生成 hash 去重。
- 设计失败重试策略与日志记录，并暴露监控指标。

### 3.3 原始内容层
- 设计 `raw_entries` ORM 模型与迁移。
- 实现原始条目查询、分页、状态更新、导出（CSV/JSON）API。
- 提供 CLI 工具导出与批量状态操作。
- 增加状态字段与索引优化查询性能。

### 3.4 筛选分析层
- 设计 `filter_rules` 模型与配置管理 API/CLI。
- 实现规则引擎：关键词、正则、评分阈值，输出 `rule_score`。
- 抽象 `LLMClient` 接口，支持多后端配置；实现基础适配器。
- 编排筛选流程：
  - 选取未处理 `raw_entries`。
  - 先运行规则过滤，通过后调用 LLM。
  - 生成结构化结果并存储至 `candidate_needs`。
- 为 LLM 调用加入超时、重试与监控指标。
- 构建 `/api/v1/filter-metrics` 指标接口，输出总体转化率、平均 `rule_score` 及按数据源拆解的命中率，供前端“筛选流程监控”模块展示。

### 3.5 候选需求层
- 设计 `candidate_needs` 模型与状态机字段。
- 实现候选需求的 REST API 与 CLI：查询、过滤、状态更新、备注。
- 构建导出/推送机制，支持 Webhook、CSV、消息队列。
- 增加状态流转日志，预留权限控制接入点。

### 3.6 调度与集成层
- 使用 APScheduler 编写周期任务：`fetch_rss`、`run_filters`、`llm_analyze`、`sync_downstream`。
- 结合 Celery Worker 承载耗时任务（LLM、批量导出）。
- 集成 Redis/RabbitMQ 作为任务队列。
- 构建 Prometheus 指标导出、Grafana 仪表板配置。
- 设置异常告警（抓取失败率、队列积压等）。

### 3.7 Web 管理前端
- **架构与环境**：
  - 选择 Vue 3 + TypeScript + Vite，集成 Element Plus（或 Ant Design Vue）、Vue Router、Vue Query/Pinia，建立 `apps/web` 或 `web` 目录。
  - 初始化 ESLint、Prettier、Stylelint、Husky/Commitlint，配置 CI 任务。
- **页面交付**：
  - 仪表盘：调用后端指标接口，绘制关键数据曲线与告警列表。
  - RSS 源管理：实现源列表、启停/频率修改、创建编辑表单、分类标签过滤。
  - 原始内容浏览：提供分页、关键字搜索、状态标记、批量忽略等操作。
  - 筛选流程监控：展示规则命中统计、LLM 成功率、耗时柱状图，支持时间过滤。
  - 候选需求工作台：支持指派负责人、状态流转、备注、批量导出 CSV。
  - 系统告警与操作日志：展示调度失败、队列积压等事件，并支持手动确认。
- **全局功能**：
  - 当前以个人使用为主暂不实现登录界面，保留 Token 登录与角色权限（只读、审核、管理员）接入点，并对无权限操作进行功能开关预留。
  - 实现国际化框架（默认中英双语），保留动态扩展能力。
  - 构建公共组件库（表格、筛选面板、图表卡片），通过 Storybook/VitePress 管理示例。
  - 引入错误边界（基于应用级异常处理）、全局加载态与通知系统，保障用户体验。
- **测试与发布**：
  - 编写 Vitest + Vue Test Utils 测试用例覆盖核心交互。
  - 使用 Playwright（可选）进行关键流程端到端测试。
  - CI 中执行 `pnpm lint`、`pnpm test`、`pnpm build`，产出可部署静态资源。

### 3.8 测试与质量保障
- 单元测试：
  - RSS 抓取器增量逻辑、去重与重试。
  - 规则引擎匹配与评分。
  - LLM 客户端接口模拟。
- 集成测试：模拟从抓取到候选需求的流水线闭环。
- 覆盖率要求 ≥ 75%，配置 coverage 报告上传。
- 静态检查：ruff、mypy、eslint、stylelint，保障类型正确性与编码规范。

### 3.9 安全、合规与运维
- 接口认证：当前部署默认关闭登录校验，但保留 Token/Key 校验、HTTPS/TLS 配置指南及启用步骤文档，便于未来扩展。
- 日志脱敏与访问控制，记录数据来源以便审计。
- 数据维护：原始内容归档策略、候选需求生命周期管理。
- 运维：定期清理任务、备份策略、容量规划建议。

## 4. 依赖与风险
- **外部依赖**：Redis/RabbitMQ、LLM 服务、Prometheus/Grafana、前端图表库（ECharts/AntV）。
- **风险点**：
  - LLM 调用稳定性与成本。
  - RSS 源质量与频率控制。
  - 多任务调度导致的资源竞争。
  - 前端与后端接口对齐、未来启用权限策略时可能出现体验不一致问题。
- **缓解措施**：
  - 对 LLM 调用实现熔断与降级策略。
  - 对 RSS 源设置采集限速与异常标记。
  - 对调度任务做优先级与资源隔离配置。
  - 制定前后端接口契约文档与集成测试，权限配置启用回滚策略。

- 引入原始条目内容指纹 `content_hash`，抓取流程可基于标题/摘要/链接自动去重，同时在 API 导出与列表接口中展示，确保重复内容不会污染后续筛选。
- 新增 API Token 认证能力，可在环境变量中配置逗号分隔的 Token，所有 `/api` 请求需通过 `X-API-Key` 或 `api_token` 校验，默认保持关闭以方便本地调试。

- 新增候选需求状态机校验能力，API 与 CLI 会在非法状态流转时返回错误，保障 `pending_review → approved/rejected → in_discovery → completed` 的流程约束。
- 新增 RSS 源抓取新鲜度告警逻辑，若启用的数据源长时间未被抓取，会在仪表盘告警接口中提示并标记严重程度，帮助快速发现调度异常。
- 新增 `.github/workflows/ci.yml`，在 Push/PR 时自动执行 `ruff check`、`pytest` 与 `mypy app/services jobs`，保证最低限度的代码质量门禁，为“监控、日志与质量保障”里程碑提前夯实 CI 能力。
- 完成持久化设施：基于 SQLAlchemy 与 Alembic 定义 `rss_sources/raw_entries/candidate_needs` 等 ORM 模型，实现 SQLite/PostgreSQL 双模支持，并在 README 中补充 `alembic upgrade head`、`docker compose up` 等操作指引，为后续任务队列、监控与前端建设奠定数据底座。
- 引入 Redis + Celery 任务队列：通过 `jobs/task_queue.py` 定义抓取、晋升、同步等任务，并以 Celery beat 或 APScheduler 扫描数据库并派发任务，Docker Compose 默认包含 `redis`、`celery worker` 与 `celery beat` 服务，确保耗时任务在独立进程异步执行并具备重试与幂等策略。
- 搭建监控与质量保障基线：新增 `app/core/metrics.py` Prometheus 导出与请求中间件，补充 RSS 抓取、候选需求晋升、下游同步及任务入队指标，`monitoring/prometheus.yml` + docker-compose `prometheus` 服务即可直接采集；`app/core/telemetry.py` 支持基于环境变量开启 OpenTelemetry Trace；提供 `python -m tools.coverage_runner` 基于 trace 模块统计覆盖率并要求 ≥ 75%，CI 上传 `coverage-summary.json`，同时在日志模块加入敏感字段脱敏策略。


## 8. 完成度评估与后续开发任务

基于当前代码仓库（仅交付 FastAPI 后端、Typer CLI、APScheduler 调度器及内存数据库）的现状，与本开发文档的目标对比仍存在以下差距，需要在后续迭代中重点补齐：

1. ✅ **数据持久化与 ORM 落地**：已完成 SQLAlchemy ORM、Alembic 初始迁移、SQLite/PG 双连接以及 Docker Compose 中的 PostgreSQL 容器，所有服务均通过 `SQLDatabase` 适配器访问真实数据库，可使用 `alembic upgrade head` 初始化数据结构。

2. ✅ **任务队列与异步执行能力**：已在 `jobs/task_queue.py` 中定义抓取、晋升、同步等 Celery 任务，新增 Redis 作为消息队列，`docker-compose` 默认包含 worker 与 beat 服务，APScheduler 也可作为手动调度入口。调度周期会扫描数据库并异步派发任务，任务执行具备重试/超时控制，并在 README 中补充了运行方式。

3. ✅ **监控、日志与质量保障体系**：API 默认挂载 `/metrics`，提供 HTTP 请求耗时与 RSS/晋升/下游/任务队列指标，`monitoring/prometheus.yml` 搭配 docker-compose 的 `prometheus` 服务即可采集；新增 `app/core/telemetry.py` 支持按需启用 OTLP 导出并为 FastAPI/Celery 注入 Trace，structlog 也实现基础脱敏；CI 运行 `ruff`、`python -m tools.coverage_runner`（基于 trace 模块统计并强制覆盖率 ≥ 75%）与 `mypy`，并上传 `coverage-summary.json`，README 同步更新运行方式。

4. ✅ **Web 管理前端与组件库**：新增 `web/` 目录并以 Vue 3 + TypeScript + Vite + Element Plus 构建管理后台骨架，接入 Pinia、Vue Router、Vue Query、Vue I18n、Vitest，完成仪表盘、RSS 源、原始内容、筛选监控、候选需求与系统告警页面的占位实现及 `StatCard`/`DataTable` 等基础组件，并在 README 中补充安装/调试指引。后续仍需：
   - 与 `/api/v1` 等真实接口打通数据流，替换示例数据源；
   - 引入 Storybook/VitePress 维护组件示例，完善国际化词条、错误边界与通知系统；
   - 在 CI 中加入 `pnpm lint/test/build`，并产出正式发布流程；
   - 根据权限策略设计 Token/角色入口与字段对齐方案。

5. **下游同步与导出通道增强**（下一阶段优先级）：现阶段仅提供 Webhook 推送，缺少 CSV/消息队列/批量导出自动化。需要：
   - 在 CLI/API 中补充 CSV/JSON 导出任务调度能力，并与任务队列联动；
   - 实现 MQ（如 Kafka/RabbitMQ）推送或文件同步策略，满足“导出/同步机制”要求；
