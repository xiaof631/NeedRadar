# 开发任务文档：NeedRadar RSS 需求挖掘系统

本任务旨在依据《开发文档：NeedRadar RSS 需求挖掘系统》实施整体研发工作，分阶段完成 RSS 采集、原始内容管理、筛选分析、候选需求管理、Web 管理前端以及调度与监控能力的建设，确保系统具备端到端的数据流转与可运维性。

## 3. 详细任务拆解
### 3.1 环境与基础设施
- 建立 FastAPI 项目结构，配置 Typer/Click 命令行框架。
- 集成 SQLAlchemy、Alembic，配置 SQLite（开发）与 PostgreSQL（生产）连接。
- 提供 Docker 与 docker-compose 模板，抽象环境变量配置。
- 集成 structlog、OpenTelemetry 基础配置。
- 建立 GitHub Actions 工作流，执行 pytest、ruff、mypy、Docker 构建。

### 3.2 RSS 源管理与采集层
- 设计 `rss_sources`、`fetch_logs` ORM 模型与迁移脚本。
- 实现 RSS 源 CRUD API 与 CLI，支持频率、分类等配置管理。
- 编写调度任务 `fetch_rss`：
  - 调度器根据频率派发抓取任务。
  - 抓取器使用 `feedparser`，支持 ETag/Last-Modified 增量抓取。
  - 写入 `raw_entries`，生成 hash 去重。
- 设计失败重试策略与日志记录，并暴露监控指标。

### 3.3 原始内容层
- 设计 `raw_entries` ORM 模型与迁移。
- 实现原始条目查询、分页、状态更新、导出（CSV/JSON）API。
- 提供 CLI 工具导出与批量状态操作。
- 增加状态字段与索引优化查询性能。

### 3.4 筛选分析层
- 设计 `filter_rules` 模型与配置管理 API/CLI。
- 实现规则引擎：关键词、正则、评分阈值，输出 `rule_score`。
- 抽象 `LLMClient` 接口，支持多后端配置；实现基础适配器。
- 编排筛选流程：
  - 选取未处理 `raw_entries`。
  - 先运行规则过滤，通过后调用 LLM。
  - 生成结构化结果并存储至 `candidate_needs`。
- 为 LLM 调用加入超时、重试与监控指标。
- 构建 `/api/v1/filter-metrics` 指标接口，输出总体转化率、平均 `rule_score` 及按数据源拆解的命中率，供前端“筛选流程监控”模块展示。

### 3.5 候选需求层
- 设计 `candidate_needs` 模型与状态机字段。
- 实现候选需求的 REST API 与 CLI：查询、过滤、状态更新、备注。
- 构建导出/推送机制，支持 Webhook、CSV、消息队列。
- 增加状态流转日志，预留权限控制接入点。

### 3.6 调度与集成层
- 使用 APScheduler 编写周期任务：`fetch_rss`、`run_filters`、`llm_analyze`、`sync_downstream`。
- 结合 Celery Worker 承载耗时任务（LLM、批量导出）。
- 集成 Redis/RabbitMQ 作为任务队列。
- 构建 Prometheus 指标导出、Grafana 仪表板配置。
- 设置异常告警（抓取失败率、队列积压等）。

### 3.7 Web 管理前端
- **架构与环境**：
  - 选择 Vue 3 + TypeScript + Vite，集成 Element Plus（或 Ant Design Vue）、Vue Router、Vue Query/Pinia，建立 `apps/web` 或 `web` 目录。
  - 初始化 ESLint、Prettier、Stylelint、Husky/Commitlint，配置 CI 任务。
- **页面交付**：
  - 仪表盘：调用后端指标接口，绘制关键数据曲线与告警列表。
  - RSS 源管理：实现源列表、启停/频率修改、创建编辑表单、分类标签过滤。
  - 原始内容浏览：提供分页、关键字搜索、状态标记、批量忽略等操作。
  - 筛选流程监控：展示规则命中统计、LLM 成功率、耗时柱状图，支持时间过滤。
  - 候选需求工作台：支持指派负责人、状态流转、备注、批量导出 CSV。
  - 系统告警与操作日志：展示调度失败、队列积压等事件，并支持手动确认。
- **全局功能**：
  - 当前以个人使用为主暂不实现登录界面，保留 Token 登录与角色权限（只读、审核、管理员）接入点，并对无权限操作进行功能开关预留。
  - 实现国际化框架（默认中英双语），保留动态扩展能力。
  - 构建公共组件库（表格、筛选面板、图表卡片），通过 Storybook/VitePress 管理示例。
  - 引入错误边界（基于应用级异常处理）、全局加载态与通知系统，保障用户体验。
- **测试与发布**：
  - 编写 Vitest + Vue Test Utils 测试用例覆盖核心交互。
  - 使用 Playwright（可选）进行关键流程端到端测试。
  - CI 中执行 `pnpm lint`、`pnpm test`、`pnpm build`，产出可部署静态资源。

### 3.8 测试与质量保障
- 单元测试：
  - RSS 抓取器增量逻辑、去重与重试。
  - 规则引擎匹配与评分。
  - LLM 客户端接口模拟。
- 集成测试：模拟从抓取到候选需求的流水线闭环。
- 覆盖率要求 ≥ 75%，配置 coverage 报告上传。
- 静态检查：ruff、mypy、eslint、stylelint，保障类型正确性与编码规范。

### 3.9 安全、合规与运维
- 接口认证：当前部署默认关闭登录校验，但保留 Token/Key 校验、HTTPS/TLS 配置指南及启用步骤文档，便于未来扩展。
- 日志脱敏与访问控制，记录数据来源以便审计。
- 数据维护：原始内容归档策略、候选需求生命周期管理。
- 运维：定期清理任务、备份策略、容量规划建议。

## 4. 依赖与风险
- **外部依赖**：Redis/RabbitMQ、LLM 服务、Prometheus/Grafana、前端图表库（ECharts/AntV）。
- **风险点**：
  - LLM 调用稳定性与成本。
  - RSS 源质量与频率控制。
  - 多任务调度导致的资源竞争。
  - 前端与后端接口对齐、未来启用权限策略时可能出现体验不一致问题。
- **缓解措施**：
  - 对 LLM 调用实现熔断与降级策略。
  - 对 RSS 源设置采集限速与异常标记。
  - 对调度任务做优先级与资源隔离配置。
  - 制定前后端接口契约文档与集成测试，权限配置启用回滚策略。

以下计划聚焦当前已经可用的 `marketplace` 线索工作台，目标是把“能抓、能看、能跟进”继续推进到“能复盘、能调优、能稳定扩源”。

##### P1 转化数据沉淀与结果闭环增强

- [x] 为 marketplace lead 增加 `lead_outcome`，支持 `won / lost / no_response / not_fit`。
- [x] 在工作台提供结果筛选、结果统计、结果时间线与结果回写。
- [x] 增加来源效果和转化复盘看板，能够按来源、按 `lead_kind / lead_tier` 观察结案率与成交率。
- [x] 为结果流转补充批量操作能力，支持在列表中批量标记 `not_fit / no_response`。
- [x] 增加“结果备注”或“失败原因标签”，把 `lost / not_fit` 再细分成可分析原因。
- [x] 为历史数据准备一次性回填脚本，方便把已有外包线索补录结果并激活复盘看板价值。

##### P2 项目画像标准化

- [ ] 增加 `budget_band` 归一化字段，例如 `lt_1k / 1k_5k / 5k_20k / gt_20k / negotiable`。
- [ ] 增加 `delivery_scope`，区分 `website / app / backend / plugin / automation / data_tool / embedded`。
- [ ] 增加 `tech_stack_normalized`，从标题、摘要、技能中归并出统一技术栈标签。
- [ ] 增加 `region / timezone_fit`，标记国内、欧美、APAC 及是否适合当前承接范围。
- [ ] 在前端补充画像筛选器，并让高纯度队列优先展示与目标承接范围更匹配的项目。

##### P3 来源运营与自动调优

- [ ] 基于 `source_breakdown + source_conversion_breakdown` 生成更强的来源策略建议，区分“继续保留 / 扩同类 / 降频 / 暂停 / 需要补结果数据”。
- [ ] 为低转化、低高纯度来源增加自动降频建议，避免持续刷入低价值线索。
- [ ] 为高纯度且有正向结果的来源增加“扩关键词 / 扩同类平台”建议，形成扩源策略库。
- [ ] 对海外职位聚合源继续压缩全职招聘噪音，优先保留 `project / contract_role`。
- [ ] 评估并接入下一批海外公开软件合同源，目标是不再主要依赖 `PeoplePerHour / Remotive / Jobicy`。

##### P4 跟进工作流增强

- [ ] 为线索详情增加“下次跟进时间”与“提醒原因”，把当前待办升级成明确的 follow-up schedule。
- [ ] 支持备注历史与结果历史并排查看，避免只看到最后一次编辑状态。
- [ ] 在列表中增加“最近操作时间 / 下次跟进时间 / 是否超时”字段，方便快速分诊。
- [ ] 为 `watching / contacted` 增加更细状态，例如 `proposal_sent / awaiting_reply / negotiation`，视真实使用情况决定是否启用。

##### P5 数据回填与运营节奏

- [x] 先从现有高纯度线索中手动补 10-20 条结果，形成第一批真实转化样本。
- [x] 每周复盘一次来源效果，记录哪些来源带来真实商机、哪些来源仅带来噪音。
- [x] 每两周评估一次过滤规则，决定是继续收紧高纯度，还是放宽扩展线索供人工筛选。

以上节奏已落到仓库：

- `docs/marketplace-ops-cadence.md`：运营节奏、回填方式、复盘流程。
- `docs/marketplace-retrospectives/`：按日期沉淀回填 CSV 和复盘记录。

##### 近期执行顺序建议

1. 先完成 P1 里剩余的“批量结果操作 + 失败原因标签 + 历史回填”，让转化看板尽快脱离全 0 状态。
2. 再推进 P2 项目画像标准化，保证后续筛选和优先级不只靠启发式标题规则。
3. 随后做 P3 来源运营，把来源调优从展示建议推进到真正指导扩源与降频。
4. 最后根据真实使用频率决定是否推进 P4 的细粒度跟进状态。
