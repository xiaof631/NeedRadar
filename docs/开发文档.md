# 开发文档：NeedRadar RSS 需求挖掘系统

## 1. 系统架构概览
系统分为六个核心子系统：
1. **采集层**：负责 RSS 源配置、增量抓取、失败重试。
2. **原始内容层**：负责存储、查询及状态标记原始条目。
3. **筛选分析层**：提供规则引擎与 LLM 结构化分析。
4. **候选需求层**：管理候选需求的生命周期与数据导出。
5. **Web 管理前端**：面向分析师与产品同学展示数据、提供可视化操作入口。
6. **调度与集成层**：调度流水线、提供监控告警、与需求挖掘系统联动。

整体采用模块化架构，每个子系统独立封装，可通过消息队列或任务调度串联；Web 前端通过 REST API 与监控接口消费后端能力。

## 2. 技术选型
- **语言与框架**：后端使用 Python 3.11（FastAPI + Typer/Click），前端使用 Vue 3 + TypeScript + Vite。
- **数据存储**：PostgreSQL（生产），SQLite（本地开发与测试）。使用 SQLAlchemy + Alembic 管理模型与迁移。
- **任务调度**：APScheduler 或 Celery Beat（结合 Redis）用于周期性抓取与处理。
- **队列系统**：Redis Stream / RabbitMQ，可用于解耦抓取、筛选、结构化处理。
- **LLM 接入**：支持本地模型（如 Llama.cpp、RWKV）或自建 API，统一通过抽象接口调用。
- **日志与监控**：使用 structlog + OpenTelemetry 导出指标，Prometheus + Grafana 可视化；前端通过 Vue Query Devtools/Vue Devtools、Sentry（可选）采集错误。
- **容器化**：使用 Docker + docker-compose 进行部署；通过环境变量配置数据源与调度参数。

## 3. 模块设计
### 3.1 RSS 源管理与采集
- 配置存储 `rss_sources`：字段包含 `id`, `name`, `url`, `category`, `frequency`, `last_fetched_at`, `status`。
- 抓取流程：
  1. 调度器根据频率推送任务。
  2. 抓取器使用 `feedparser` 解析 RSS，支持 ETag/Last-Modified 实现增量。
  3. 将条目写入 `raw_entries` 表，附带源 ID、hash 值用于去重。
- 对失败任务写入 `fetch_logs` 表，并触发重试策略。

### 3.2 原始内容库
- `raw_entries` 表字段：`id`, `source_id`, `title`, `summary`, `content`, `link`, `published_at`, `author`, `tags`, `status`, `created_at`, `updated_at`。
- 提供服务：
  - 列表/分页查询 API。
  - 状态更新 API（标记为忽略、已处理等）。
  - 导出 API（CSV/JSON）。

### 3.3 筛选分析层
- 规则引擎：
  - 配置表 `filter_rules` 定义关键词、正则、评分阈值等。
  - 引擎逐条匹配，计算得分并标记 `rule_score`。
- LLM 分析：
  - 定义抽象接口 `LLMClient`，实现 `analyze_entry(entry) -> StructuredNeed`。
  - 支持不同模型后端（本地 GGML、开源 API）。
  - LLM 输出字段：`problem_statement`, `target_users`, `value_proposition`, `competition`, `confidence`。
- 筛选流程：
  1. 选取状态为“未处理”的 `raw_entries`。
  2. 先执行规则过滤，若通过则进入 LLM 分析（可选）。
  3. 生成 `candidate_needs` 记录并更新原始条目状态。

### 3.4 候选需求管理
- `candidate_needs` 表字段：`id`, `raw_entry_id`, `summary`, `problem_statement`, `target_users`, `value`, `competition`, `confidence`, `status`, `created_at`, `updated_at`。
- 支持状态机：`pending_review` → `approved`/`rejected` → `in_discovery` → `completed`。
- 提供 REST API 与 CLI：
  - 列表查询、过滤、导出。
  - 状态更新、备注补充。
  - 推送至消息队列或 Webhook。

### 3.5 调度与监控
- 使用 APScheduler 配置定时任务：`fetch_rss`, `run_filters`, `llm_analyze`, `sync_downstream`。
- 引入 Celery 以分布式运行繁重任务（LLM 调用、批量导出）。
- 监控指标：
  - 采集：成功率、抓取时长、条目数。
  - 筛选：通过率、LLM 延迟、得分分布。
  - 候选需求：入库量、状态转化率。
- 告警：对抓取失败率过高、队列积压等情况触发通知（邮件/Slack/Webhook）。

### 3.6 Web 管理前端
- **技术选型**：Vue 3 + TypeScript + Vite 作为基础构建工具，UI 组件库选用 Element Plus（或 Ant Design Vue），使用 Vue Query/自封装 hooks 管理数据请求状态。
- **页面结构**：
  - 仪表盘：展示抓取量、筛选通过率、候选需求转化率、调度任务健康度等核心指标，支持时间区间筛选。
  - RSS 源管理：列表展示源配置，支持启停、频率调整、分类标签维护，并提供新增/编辑弹窗。
  - 原始内容浏览：以表格或卡片形式呈现 `raw_entries`，支持关键字搜索、状态过滤、快速标记忽略。
  - 筛选流程监控：可视化每个规则与 LLM 分析的通过率、耗时分布，并可查看失败原因。
  - 候选需求工作台：支持分配责任人、批量状态变更、备注记录以及导出操作。
  - 系统告警与审计：集中展示调度失败、队列积压等告警，并提供操作日志查询。
- **全局能力**：
  - 目前作为个人使用场景默认免登录，保留统一身份认证（后端 Token）接入点及角色权限扩展接口，后续可按只读/审核/管理员等角色落地。
  - 通过国际化方案（i18n）支持多语言扩展，默认提供中英文文案。
  - 与 Prometheus/Grafana 指标联动，嵌入关键图表或提供外链。
  - 为 RSS 源、候选需求列表提供客户端缓存与增量刷新，减少 API 压力。

### 3.7 开发规范与交互约束（前后端）
- REST API 返回统一的 `code/message/data` 结构，错误信息在前端 toast/notification 呈现。
- 所有变更操作需在前端二次确认，并记录操作者与时间戳。
- 前端表单字段与后端模型字段保持映射文档，防止命名不一致。
- 通过 Storybook/Playroom 维护关键组件的交互示例，保障 UI 复用与回归。

### 3.8 测试与质量保障
- 单元测试：
  - RSS 抓取器增量逻辑、去重与重试。
  - 规则引擎匹配与评分。
  - LLM 客户端接口模拟。
- 集成测试：模拟从抓取到候选需求的流水线闭环。
- 前端测试：使用 Vue Test Utils + Vitest 覆盖核心页面交互、状态管理逻辑。
- 覆盖率要求 ≥ 75%，配置 coverage 报告上传。
- 静态检查：ruff、mypy、eslint、stylelint，保障类型正确性与编码规范。

### 3.9 安全、合规与运维
- 接口认证：当前个人部署默认关闭登录校验，但保留 Token/Key 校验与 HTTPS/TLS 配置指南，便于未来启用。
- 日志脱敏与访问控制，记录数据来源以便审计。
- 数据维护：原始内容归档策略、候选需求生命周期管理。
- 运维：定期清理任务、备份策略、容量规划建议。

## 4. 开发流程
1. **环境准备**：
   - 克隆代码、创建虚拟环境、安装依赖。
   - 使用 `.env` 配置数据库、消息队列连接。
2. **数据库迁移**：使用 Alembic 生成迁移脚本，执行 `alembic upgrade head`。
3. **本地运行**：
   - 运行 `uvicorn app.main:app --reload` 启动 API。
   - 运行 `python -m jobs.scheduler` 启动调度器。
   - 在 `web` 目录执行 `pnpm install && pnpm dev` 启动管理前端，默认监听 5173 端口。
4. **测试策略**：
   - 单元测试覆盖数据抓取、规则过滤、LLM 接口。
   - 集成测试模拟完整流水线。
   - 使用 pytest + coverage，目标覆盖率 75%+。
5. **CI/CD**：
   - GitHub Actions：运行测试、静态检查（ruff/mypy、eslint、stylelint）、构建 Docker 镜像与前端静态资源。
   - 发布流程：通过版本标签触发部署，使用 docker-compose 在服务器拉起服务，同时在 CI 构建前端产物并上传至对象存储或 Nginx 静态目录。

## 5. 配置与部署
- **配置文件**：`config.yaml` 定义 RSS 源、规则、任务频率；可通过环境变量覆盖。
- **部署建议**：
  - 使用 docker-compose 编排（web、worker、scheduler、db、redis）。
  - 持久化存储映射至主机磁盘。
  - 使用 Traefik/Nginx 提供统一入口与 TLS 终端。
- **扩展插件**：
  - RSS 源适配器：实现 `BaseSourceAdapter` 接口以支持自定义数据源。
  - 分析插件：实现 `BaseAnalyzer` 接口接入新的评分策略。

## 6. 数据流与序列图
1. 调度器触发 `fetch_rss` → 抓取器读取 `rss_sources` → 调用 adapter 获取数据 → 存储至 `raw_entries`。
2. `run_filters` 从 `raw_entries` 读取未处理数据 → 应用规则 → 触发 `llm_analyze` → 写入 `candidate_needs`。
3. `sync_downstream` 读取新增候选需求 → 通过 Webhook/文件导出 → 下游需求挖掘系统消费。
4. Web 管理前端通过 REST/GraphQL API 拉取数据 → 展示仪表盘与工作台 → 用户操作通过 API 回写状态并触发审计日志。

## 7. 安全与合规
- 仅加载公开 RSS 源，记录数据来源以便合规审计。
- 对外 API 需 Token 鉴权，采用 HTTPS 传输。
- 日志中避免记录敏感信息，遵循最小权限原则。

## 8. 运维与未来优化
- 定期清理过期原始内容、归档历史需求。
- 加入自动化关键词挖掘与趋势分析算法。
- 持续扩展前端仪表盘，支持自定义报表、拖拽布局与实时协作批注。
