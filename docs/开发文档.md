# 开发文档：NeedRadar RSS 需求挖掘系统

## 1. 系统架构概览
系统分为五个核心子系统：
1. **采集层**：负责 RSS 源配置、增量抓取、失败重试。
2. **原始内容层**：负责存储、查询及状态标记原始条目。
3. **筛选分析层**：提供规则引擎与 LLM 结构化分析。
4. **候选需求层**：管理候选需求的生命周期与数据导出。
5. **调度与集成层**：调度流水线、提供监控告警、与需求挖掘系统联动。

整体采用模块化架构，每个子系统独立封装，可通过消息队列或任务调度串联。

## 2. 技术选型
- **语言与框架**：Python 3.11，使用 FastAPI 构建服务端接口，Typer/Click 提供命令行工具。
- **数据存储**：PostgreSQL（生产），SQLite（本地开发与测试）。使用 SQLAlchemy + Alembic 管理模型与迁移。
- **任务调度**：APScheduler 或 Celery Beat（结合 Redis）用于周期性抓取与处理。
- **队列系统**：Redis Stream / RabbitMQ，可用于解耦抓取、筛选、结构化处理。
- **LLM 接入**：支持本地模型（如 Llama.cpp、RWKV）或自建 API，统一通过抽象接口调用。
- **日志与监控**：使用 structlog + OpenTelemetry 导出指标，Prometheus + Grafana 可视化。
- **容器化**：使用 Docker + docker-compose 进行部署；通过环境变量配置数据源与调度参数。

## 3. 模块设计
### 3.1 RSS 源管理与采集
- 配置存储 `rss_sources`：字段包含 `id`, `name`, `url`, `category`, `frequency`, `last_fetched_at`, `status`。
- 抓取流程：
  1. 调度器根据频率推送任务。
  2. 抓取器使用 `feedparser` 解析 RSS，支持 ETag/Last-Modified 实现增量。
  3. 将条目写入 `raw_entries` 表，附带源 ID、hash 值用于去重。
- 对失败任务写入 `fetch_logs` 表，并触发重试策略。

### 3.2 原始内容库
- `raw_entries` 表字段：`id`, `source_id`, `title`, `summary`, `content`, `link`, `published_at`, `author`, `tags`, `status`, `created_at`, `updated_at`。
- 提供服务：
  - 列表/分页查询 API。
  - 状态更新 API（标记为忽略、已处理等）。
  - 导出 API（CSV/JSON）。

### 3.3 筛选分析层
- 规则引擎：
  - 配置表 `filter_rules` 定义关键词、正则、评分阈值等。
  - 引擎逐条匹配，计算得分并标记 `rule_score`。
- LLM 分析：
  - 定义抽象接口 `LLMClient`，实现 `analyze_entry(entry) -> StructuredNeed`。
  - 支持不同模型后端（本地 GGML、开源 API）。
  - LLM 输出字段：`problem_statement`, `target_users`, `value_proposition`, `competition`, `confidence`。
- 筛选流程：
  1. 选取状态为“未处理”的 `raw_entries`。
  2. 先执行规则过滤，若通过则进入 LLM 分析（可选）。
  3. 生成 `candidate_needs` 记录并更新原始条目状态。

### 3.4 候选需求管理
- `candidate_needs` 表字段：`id`, `raw_entry_id`, `summary`, `problem_statement`, `target_users`, `value`, `competition`, `confidence`, `status`, `created_at`, `updated_at`。
- 支持状态机：`pending_review` → `approved`/`rejected` → `in_discovery` → `completed`。
- 提供 REST API 与 CLI：
  - 列表查询、过滤、导出。
  - 状态更新、备注补充。
  - 推送至消息队列或 Webhook。

### 3.5 调度与监控
- 使用 APScheduler 配置定时任务：`fetch_rss`, `run_filters`, `llm_analyze`, `sync_downstream`。
- 引入 Celery 以分布式运行繁重任务（LLM 调用、批量导出）。
- 监控指标：
  - 采集：成功率、抓取时长、条目数。
  - 筛选：通过率、LLM 延迟、得分分布。
  - 候选需求：入库量、状态转化率。
- 告警：对抓取失败率过高、队列积压等情况触发通知（邮件/Slack/Webhook）。

## 4. 开发流程
1. **环境准备**：
   - 克隆代码、创建虚拟环境、安装依赖。
   - 使用 `.env` 配置数据库、消息队列连接。
2. **数据库迁移**：使用 Alembic 生成迁移脚本，执行 `alembic upgrade head`。
3. **本地运行**：
   - 运行 `uvicorn app.main:app --reload` 启动 API。
   - 运行 `python -m jobs.scheduler` 启动调度器。
4. **测试策略**：
   - 单元测试覆盖数据抓取、规则过滤、LLM 接口。
   - 集成测试模拟完整流水线。
   - 使用 pytest + coverage，目标覆盖率 75%+。
5. **CI/CD**：
   - GitHub Actions：运行测试、静态检查（ruff/mypy）、构建 Docker 镜像。
   - 发布流程：通过版本标签触发部署，使用 docker-compose 在服务器拉起服务。

## 5. 配置与部署
- **配置文件**：`config.yaml` 定义 RSS 源、规则、任务频率；可通过环境变量覆盖。
- **部署建议**：
  - 使用 docker-compose 编排（web、worker、scheduler、db、redis）。
  - 持久化存储映射至主机磁盘。
  - 使用 Traefik/Nginx 提供统一入口与 TLS 终端。
- **扩展插件**：
  - RSS 源适配器：实现 `BaseSourceAdapter` 接口以支持自定义数据源。
  - 分析插件：实现 `BaseAnalyzer` 接口接入新的评分策略。

## 6. 数据流与序列图
1. 调度器触发 `fetch_rss` → 抓取器读取 `rss_sources` → 调用 adapter 获取数据 → 存储至 `raw_entries`。
2. `run_filters` 从 `raw_entries` 读取未处理数据 → 应用规则 → 触发 `llm_analyze` → 写入 `candidate_needs`。
3. `sync_downstream` 读取新增候选需求 → 通过 Webhook/文件导出 → 下游需求挖掘系统消费。

## 7. 安全与合规
- 仅加载公开 RSS 源，记录数据来源以便合规审计。
- 对外 API 需 Token 鉴权，采用 HTTPS 传输。
- 日志中避免记录敏感信息，遵循最小权限原则。

## 8. 运维与未来优化
- 定期清理过期原始内容、归档历史需求。
- 加入自动化关键词挖掘与趋势分析算法。
